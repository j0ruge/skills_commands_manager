# Tenant ID ↔ Zitadel Org ID Mapping

This reference is short on purpose: it's a single concept that bites every project once.

## The problem

Your domain layer has a stable `tenantId` value object — typically a short uppercase string like `"JRC"` or `"ACME"`. That string is what your code passes around, what shows in logs, what your tests assert on.

Zitadel's API expects the **numeric `orgId`** in the `x-zitadel-orgid` header on every Management API call:

```http
x-zitadel-orgid: 370503937624637443
```

`tenantId.toString()` returns `"JRC"` and Zitadel rejects it with `403 Organisation doesn't exist (AUTH-Bs7Ds)`.

## Why both exist

- **Domain `tenantId`** is what the business cares about — it's stable across IdP migrations. If you swap Zitadel for Authentik in 5 years, your `tenantId` stays the same, your code doesn't change.
- **Zitadel `orgId`** is an internal identifier generated by Zitadel at org creation. It's numeric, opaque, and instance-specific. If you wipe and recreate the org, the `orgId` changes.

The boundary between these two lives in your **infrastructure adapter**. Domain code should never see numeric orgIds; the adapter translates on every call.

## Recommended pattern

In your IdP adapter, accept a tenant→orgId mapping at construction:

```typescript
// infrastructure/adapters/zitadel/config.ts
const schema = z.object({
  ZITADEL_API_URL: z.string().url(),
  OIDC_ISSUER: z.string().url(),
  OIDC_AUDIENCE: z.string().min(1),
  ZITADEL_PROJECT_ID: z.string().min(1),
  // Mapping. One env var per tenant, looked up dynamically.
  // E.g., ZITADEL_ORG_ID_JRC=370503937624637443
});

// infrastructure/adapters/zitadel/ZitadelIdentityProvider.ts
export class ZitadelIdentityProvider {
  constructor(
    private readonly client: ZitadelClient,
    private readonly config: ZitadelConfig,
  ) {}

  // Translate domain tenant → numeric orgId on every API call.
  private resolveOrgId(tenantId: TenantId): string {
    const tenant = tenantId.toString();
    const envKey = `ZITADEL_ORG_ID_${tenant}` as keyof ZitadelConfig;
    const mapped = this.config[envKey];
    if (typeof mapped === 'string' && mapped.length > 0) return mapped;
    // Fallback: assume tenantId is already numeric (single-tenant deployment).
    return tenant;
  }

  async deactivateUser(zitadelUserId: ZitadelUserId, tenantId: TenantId) {
    await this.client.management(`/management/v1/users/${zitadelUserId}/_deactivate`, {
      method: 'POST',
      body: '{}',
      headers: { 'x-zitadel-orgid': this.resolveOrgId(tenantId) },
    });
  }
}
```

Why env vars per tenant rather than a JSON map: env vars survive container restarts, are easy to audit in deploy pipelines, and don't require a config file mount. For >5 tenants, switch to a JSON config file.

## Where the mapping comes from

The `bootstrap-zitadel.ts` script writes a JSON file with the mapping after first boot:

```json
{
  "orgId": "370503937624637443",
  "orgName": "JRC",
  "projectId": "370503956331233283",
  "appId": "370503965692919811",
  "clientId": "370503965692985347",
  "redirectUris": ["..."],
  "roles": ["battery.operator", "battery.admin"]
}
```

In production, copy `orgId` → `ZITADEL_ORG_ID_<TENANT>` in your secret manager.

## Ports vs adapters

In a hexagonal architecture, the **port** (`IdentityProvider` interface) takes domain types:

```typescript
interface IdentityProvider {
  deactivateUser(zitadelUserId: ZitadelUserId, tenantId: TenantId): Promise<void>;
  // tenantId is the DOMAIN type, not the Zitadel orgId
}
```

The **adapter** does the translation. The port is stable; if you swap the IdP, only the adapter changes — `tenantId` stays in the domain language.

## Mapping the inverse direction (orgId → tenantId)

Your JWT validation middleware extracts `urn:zitadel:iam:org:id` (numeric orgId) from the token and needs to map it back to the domain tenantId for the `AuthContext`. Build a reverse lookup at boot:

```typescript
function buildOrgIdToTenantMap(env: NodeJS.ProcessEnv): Map<string, string> {
  const map = new Map<string, string>();
  for (const [key, value] of Object.entries(env)) {
    if (key.startsWith('ZITADEL_ORG_ID_') && value) {
      const tenant = key.slice('ZITADEL_ORG_ID_'.length);
      map.set(value, tenant);
    }
  }
  return map;
}

function domainTenantFromOrgId(orgId: string): string {
  const tenant = ORG_ID_TO_TENANT.get(orgId);
  return tenant ?? orgId;  // fallback for unknown orgs
}
```

The fallback uses the orgId itself when no mapping is configured — useful in dev / single-tenant.

## ACL parity: backend ↔ frontend

If your backend translates **human** IdP roles (`battery.operator`, `battery.admin`) into **canonical** domain roles (`battery:writer`, `battery:reader`, `battery:admin`) via an Anti-Corruption Layer, **the SPA must duplicate that translation table**. Reason: in the SPA + API + IdP triangle, the access token goes **directly** from IdP to SPA. The backend never gets to inspect or rewrite it before the SPA reads `roles` from the JWT for UI gating.

**Symptom of a missing parity** (a real one we hit): a user with Zitadel role `battery.admin` logs in. Backend-protected routes work fine (backend translates the claim and authorizes). But the UI hides every admin button because the SPA's `extractRoles` filters for `["battery:reader", "battery:writer", "battery:admin"]` and the JWT only has `battery.admin` (with a dot, not a colon).

```typescript
// SPA must mirror the backend ROLE_TRANSLATION_TABLE:
const ROLE_TRANSLATION = {
  'battery.operator': ['battery:writer', 'battery:reader'],
  'battery.admin':    ['battery:admin', 'battery:writer', 'battery:reader'],
} as const;

function extractRoles(profile: IdTokenClaims): BatteryRole[] {
  const claim = profile['urn:zitadel:iam:org:project:roles'] ?? {};
  const raw = Array.isArray(claim) ? claim : Object.keys(claim);
  const resolved = new Set<BatteryRole>();
  for (const r of raw) {
    if (CANONICAL_ROLES.includes(r)) resolved.add(r as BatteryRole);
    ROLE_TRANSLATION[r]?.forEach((m) => resolved.add(m));
  }
  return [...resolved];
}
```

**Principle**: whenever a token bypasses an interpreter, the interpretation must be re-applied at every consumer. Either translate at the IdP (Zitadel Actions can rewrite claims pre-emission) **or** translate at every consumer (backend AND frontend), but never only at one consumer downstream.

For the JRC ERP we chose option B (translate at consumers) because it keeps the IdP config minimal and auditable; the trade-off is that any new consumer (mobile app, CLI, batch job) must include the same translation table.

## Future-proofing

If your project will likely have many tenants, model this as a first-class **TenantRegistry** port:

```typescript
interface TenantRegistry {
  resolveOrgId(tenantId: TenantId): Promise<string>;
  resolveTenantId(orgId: string): Promise<TenantId>;
}
```

Implementation can be env-driven (V0), then DB-backed when you onboard real tenants. The IdP adapter consumes the port instead of reading env directly. This is overkill for 1-3 tenants but pays off at scale.
